## グランドルール（厳守）

### TDD（テスト駆動開発）を強制する

Kent Beck 氏や t_wada 氏が提唱する TDD 開発フローに従うことを絶対条件とします。

#### Red → Green → Refactor サイクル

すべての機能開発は以下のサイクルで進める：

1. **Red（赤）** — テストを先に書く
   - [ ] 失敗するテストコードを最初に書く
   - 本体コードはまだ書かない
   - テストが失敗することを確認する

2. **Green（緑）** — テストを通す
   - [ ] テストを通すための最小限の本体コードを作成
   - 完璧さは求めない、まずは成功させる
   - テストがすべて通ることを確認する

3. **Refactor（リファクタ）** — コードを整理
   - [ ] テストを通したまま、本体コードをリファクタリング
   - 可読性、保守性を向上させる
   - テストが引き続き通ることを確認する

### タスク管理

すべてのタスクをチェックボックス形式で管理する：

```
- [ ] タスク内容（未着手）
- [x] 完了したタスク
```

---

## 開発ルール

- **テストなしに本体コードを書かない**
  - Red フェーズなしに Green フェーズに進むことは禁止
  
- **グランドルールを破ってはいけない**
  - TDD のサイクルは絶対条件
  - 「急いでるから」「簡単だから」という理由での例外は認めない

- **テストの完全性を確保**
  - テストコードも本体コード同様に、品質を維持する
  - 不完全なテストは本体コードの品質を保証できない

## 開発フロー例

```
## 機能 X を追加する

### Red（テストを先に書く）
- [ ] テストケース 1 を書く（失敗することを確認）
- [ ] テストケース 2 を書く（失敗することを確認）

### Green（テストを通す）
- [ ] 機能 X の最小実装を完成させる
- [ ] すべてのテストが通ることを確認

### Refactor（コードを整理）
- [ ] コードの可読性を向上させる
- [ ] 重複を削除する
- [ ] テストが引き続き通ることを確認
```
